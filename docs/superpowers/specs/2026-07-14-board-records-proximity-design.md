# 棋盘与记录区邻近布局设计

## 问题与目标

活动页当前使用“游戏区 + 侧栏”的双列 Grid，而机会明细和中奖列表位于整个 Grid 之后。右侧任务与排行榜高于棋盘时，会撑高 Grid 首行，导致记录区与棋盘之间出现大段无意义空白。

目标是让“走棋 → 查看机会变化或中奖结果”形成连续阅读路径，同时不压缩排行榜、不增加抽屉操作，也不改变记录数据与加载逻辑。

## 页面结构

- 新增左侧连续内容列 `game-column`，内部依次放置原 `game-area` 和 `records`。
- `dashboard` 只保留两个直接子项：`game-column` 与现有 `side-panel`。
- PC 端左列与右侧侧栏顶端对齐，但两列高度互不决定内部内容位置。
- `records` 紧跟棋盘，间距使用 16px，并取消原先独立的页面级最大宽度、左右 padding 和大底部定位。
- PC 端机会明细和中奖列表保持两列等宽，与棋盘左右边界严格对齐。

## 响应式顺序

宽屏顺序：

1. 左列：状态卡、棋盘、机会明细与中奖列表。
2. 右列：幸运中心入口、今日任务与排行榜。

手机及窄屏顺序：

1. 状态卡。
2. 棋盘。
3. 机会明细。
4. 中奖列表。
5. 幸运中心入口与今日任务。
6. 排行榜。

手机端记录列表改为单列，棋盘与记录区间距 12px；原有默认展开、每页 10 条、继续加载和错误重试规则不变。

## 视觉与行为约束

- 不缩短排行榜、不在排行榜内部增加滚动条。
- 不把记录区做成抽屉、浮层或覆盖棋盘的元素。
- 不修改棋盘尺寸、骰子区域、任务内容、排行榜内容或后端查询。
- 使用现有 4px 间距体系中的 12px/16px，不新增任意间距值。
- 记录卡片视觉样式保持一致，避免布局调整同时引入无关重新设计。

## 测试与验收

- 页面测试确认 `records` 位于 `game-column` 内，且 `side-panel` 是其兄弟节点。
- CSS/静态检查确认左列使用纵向布局、记录区取消页面级 padding，并在窄屏切换为单列。
- 回归运行完整 PHP、Node、Vite 构建和界面布局扫描。
- 人工结构验收确认 PC 端记录紧随棋盘，手机端记录出现在任务和排行榜之前。
