# 幸运跳棋大冒险活动设计方案

> 版本：V2.0
> 技术方向：PHP 8.2+ + Laravel 12 + SQLite
> 文档用途：产品原型、UI 设计、后端开发、测试验收及活动运营

## 1. 活动概述

用户注册并登录后，通过连续签到、本人充值、邀请好友注册及好友首次充值达标等任务获得“跳棋机会”。每消耗 1 次机会，可在 36 格棋盘上随机前进，并触发所在格子的奖品或特殊事件。

活动期间，系统根据用户完成圈数、当前格子数和到达当前进度的时间生成实时排行榜。排行榜展示前 20 名，活动结束并完成风控审核后，第 1～10 名获得排名奖励。

## 2. 活动目标

- 提高用户每日登录和连续签到率。
- 促进用户充值和复充。
- 通过邀请任务带来新注册用户。
- 通过好友首充任务提升邀请用户的付费转化。
- 使用棋盘、奖品、事件和排行榜增强活动趣味性与竞争性。
- 通过每日/每周任务、圈数宝箱、道具、成就和皮肤建立长期成长目标。
- 通过中奖播报、领奖进度和消息中心增强活动可信度。

## 3. 基础参与规则

- 用户必须注册并登录后才能参加活动。
- 同一用户只能使用一个活动账户。
- 只有活动有效期内产生的签到、充值和邀请行为才计入任务。
- 所有日期与时间以服务器配置的活动时区为准。
- 每次机会获得、消耗、补发和扣回均须生成不可篡改的业务流水。
- 后台可以设置活动开始时间、结束时间、排行榜冻结时间和领奖截止时间。
- 中奖及排名奖励发放前，平台有权进行风控审核。

## 4. 跳棋机会获取规则

### 4.1 连续签到

普通签到每天获得 **5 次跳棋机会**。

| 连续签到天数 | 当天获得机会 |
| --- | ---: |
| 第 1～6 天 | 5 次 |
| 第 7 天 | 10 次 |
| 第 8～13 天 | 5 次 |
| 第 14 天 | 10 次 |
| 第 15 天及以后 | 按 7 天周期循环；第 21、28 天等获得 10 次 |

规则说明：

- 每个自然日只能签到一次，不支持补签。
- 第 7、14、21、28 天的 10 次为当天总奖励，不是额外增加 10 次。
- 如果中断签到，连续天数清零；下次签到从第 1 天重新计算，并获得 5 次。
- 连续签到以活动指定时区的自然日判断，不能简单按间隔 24 小时判断。

### 4.2 本人充值

用户每累计充值 **10 USDT，获得 10 次跳棋机会**，可以重复获得。

计算原则：

```text
本次应发达标份数 = floor(累计有效充值金额 / 10 USDT) - 已发放达标份数
本次发放机会 = 本次应发达标份数 × 10
```

示例：

| 充值情况 | 获得机会 |
| --- | ---: |
| 一次充值 10U | 10 次 |
| 一次充值 25U | 20 次，剩余 5U 继续累计 |
| 后续再充值 5U | 10 次 |
| 一次充值 100U | 100 次 |

补充规则：

- 只有状态为“支付成功”的充值订单参与活动。
- 外部充值订单号必须唯一，重复回调不能重复发放。
- 充值撤销、退款或被认定为风险订单时，应扣回对应机会并生成流水。
- 如果支持其他币种，应按照充值确认时的后台汇率折算为 USDT，并保存当时汇率。
- 后台可以配置个人每日或整个活动期间的充值奖励上限。

### 4.3 邀请好友注册

每成功邀请一名有效好友注册，邀请人获得 **5 次跳棋机会**。

有效邀请条件：

- 好友通过邀请链接或邀请码注册。
- 好友完成手机号或邮箱验证。
- 被邀请人与邀请人不能是同一用户。
- 每个好友只能绑定一个邀请人，绑定后不能自行修改。
- 同一好友只能给邀请人发放一次注册奖励。
- 系统可结合设备、IP、联系方式和支付信息识别批量注册行为。

### 4.4 好友首次充值达标

被邀请好友累计有效充值首次达到 **10 USDT** 后：

- 邀请人获得 **10 次跳棋机会**。
- 每名好友只触发一次。
- 好友后续再次充值，不再给邀请人赠送机会。
- 好友本人仍按“本人充值”规则获得自己的跳棋机会。

例如：好友第一次充值 6U 时暂不发放；第二次充值 4U 后累计达到 10U，邀请人获得 10 次；好友以后继续充值，不再产生邀请人奖励。

## 5. 36 格大富翁棋盘设计

棋盘虽然承载跳棋玩法，但视觉形式采用经典“大富翁”设计：36 个格子首尾相连，沿矩形或圆角方形外围组成一圈，中间区域展示活动主题、剩余机会、骰子按钮和用户信息。格子不全部放置奖品，以普通格、移动事件格、趣味事件格和少量奖励格共同构成游戏节奏。

### 5.1 行走规则

- 棋盘共有 36 个格子，用户初始位于第 0 格起点。
- 每次正常跳棋消耗 1 次机会。
- 点数由 PHP 服务端使用安全随机数生成，建议范围为 1～6。
- 前端只播放动画，不能提交或决定点数、奖品及事件结果。
- 用户超过第 36 格后进入下一圈，剩余步数继续计算。

示例：用户位于第 34 格，抽到 5 点，则完成一圈并停在下一圈第 3 格。

累计进度计算方式：

```text
累计进度 = 已完成圈数 × 36 + 当前格子数
```

### 5.2 建议格子配置

| 格子类型 | 建议数量 | 说明 |
| --- | ---: | --- |
| 普通格 | 14 | 无奖励，可显示鼓励、剧情或品牌文案 |
| 前进 3 格 | 3 | 落地后额外前进 3 格 |
| 前进 5 格 | 2 | 落地后额外前进 5 格 |
| 后退 2 格 | 3 | 落地后退 2 格，最低退到当前圈起点 |
| 电池 | 3 | 发放业务电池奖励 |
| VIP +1 | 1 | VIP 等级提升 1 级，具体上限按业务规则处理 |
| 0.01 USDT | 1 | 发放小额 USDT 奖励 |
| 50 PHP | 1 | 发放 50 菲律宾比索奖励 |
| 10000 VND | 1 | 发放 10000 越南盾奖励 |
| 冰冻 | 3 | 下次需要消耗 1 次机会解冻 |
| 炸弹 | 2 | 回到当前圈起点 |
| 额外机会 | 2 | 获得后台配置数量的额外机会 |
| **合计** | **36** | 具体格子位置由后台配置 |

运营配置建议：

- 现金、VIP、电池等奖励格只占少部分，不把棋盘全部填成奖品格。
- 普通格、移动格、奖励格、冰冻格和炸弹格应交错分布。
- 避免冰冻和炸弹连续出现，减少过强的挫败感。
- 前进格后方不紧接炸弹，后退格前方不紧接冰冻，避免事件组合过于惩罚用户。
- 每类奖品可以配置活动总库存、每日库存和个人中奖上限。
- 修改棋盘配置只影响后续移动记录，历史记录保留当时的格子配置快照。

### 5.2.1 地标格与安全格补充设计

原普通文案格不再全部视为“无意义格子”。其中 12 格设为紫色地标格，覆盖旅行驿站、阳光大道、彩虹街区、欢乐车站、幸运转角、星光广场、海滨大道、音乐小镇、森林公园、梦想港湾、金色大道和终点冲刺。

- 地标首次到达解锁永久印章，并触发对应轻量效果。
- 重复到达转化为幸运值，同时保留该地标效果。
- 收集 3、6、9、12 个不同地标可领取一次性阶段宝箱。
- 其余普通格明确标记为安全格，用于控制奖励密度和负面事件频率，落地时提示“安全抵达”。
- 棋盘通过五色图例区分安全、地标、增益、风险和奖励；点击或悬停格子可查看详细说明。
- 首次访问提供五步新手引导，并提供独立玩法帮助、地标图鉴和分类 FAQ 页面。

### 5.3 大富翁视觉与交互设计

整体风格：

- 采用大富翁式环形棋盘，每个格子有独立颜色、图标、名称和边框。
- 四个角格尺寸可以略大，其中一个角为起点；其余角格用于重点事件或活动主题装饰。
- 普通格使用较低饱和度背景，奖品格使用金色或高亮色，移动格使用箭头，冰冻和炸弹使用明确图标。
- 当前用户棋子始终高亮，并通过发光、呼吸或轻微浮动效果与其他元素区分。
- 掷骰后按经过格子逐格播放移动动画，最终再展示奖励或事件弹窗。
- 棋盘中央区域显示活动标题、骰子、剩余机会、当前圈数和“立即跳棋”按钮。
- 排行榜、任务列表、机会明细和中奖列表放在棋盘周围或通过抽屉、弹窗打开，不遮挡主要移动路线。

信息层级：

```text
第一层：棋盘、棋子、骰子、剩余机会、跳棋按钮
第二层：当前圈数、当前排名、任务入口、排行榜入口
第三层：活动规则、机会流水、中奖记录、邀请详情
```

### 5.4 PC 端适配

- 内容区建议最大宽度为 1200～1440px，居中展示。
- 棋盘位于页面左侧或中央，占主要视觉区域；右侧可固定展示任务列表与实时排行榜。
- 棋盘建议保持接近 1:1 的比例，避免窗口拉伸导致格子变形。
- 36 格沿外围排列，中间保留足够空间放置骰子和主要按钮。
- 浏览器宽度下降时，右侧面板移到棋盘下方，不缩小到无法识别格子文字。
- 鼠标悬停格子时可以展示完整奖品或事件说明，但重要信息不能只依赖悬停。

推荐布局：

```text
┌───────────────────────────────┬──────────────┐
│                               │ 用户状态     │
│      36 格大富翁棋盘          │ 任务列表     │
│      中央为骰子和按钮         │ 实时排行榜   │
│                               │              │
└───────────────────────────────┴──────────────┘
│ 机会明细 / 中奖记录 / 活动规则               │
└──────────────────────────────────────────────┘
```

### 5.5 手机端适配

- 采用移动端优先设计，支持常见 360px～430px 屏幕宽度。
- 顶部紧凑展示活动倒计时、剩余机会、圈数和当前排名。
- 棋盘按屏幕宽度等比缩放并保持完整可见，不能要求用户横向滚动才能看到整圈。
- 格子文字空间不足时显示图标和短名称，点击格子后通过浮层显示完整说明。
- 骰子和主要按钮放在棋盘中央，点击区域不小于 44×44px。
- 任务、榜单、机会和中奖记录采用底部标签栏或底部抽屉展示。
- 弹窗高度不能超过可视区域，内容较多时弹窗内部滚动。
- 兼容刘海屏和底部安全区，底部按钮使用 `safe-area-inset-bottom` 留白。
- 移动动画应控制时长并支持低性能设备；用户关闭动画后可以直接显示结果。

推荐布局：

```text
┌──────────────────────┐
│ 倒计时  机会  圈数  排名 │
├──────────────────────┤
│                      │
│   36 格大富翁棋盘     │
│   中央骰子与主按钮     │
│                      │
├──────────────────────┤
│ 任务  榜单  明细  中奖 │
└──────────────────────┘
```

## 6. 特殊格子规则

### 6.1 冰冻

- 用户落在冰冻格后，活动状态变为“已冰冻”。
- 用户下次点击按钮时，消耗 1 次跳棋机会完成解冻，本次不会前进。
- 解冻后用户仍停留在原格，但不会再次触发该格的冰冻事件。
- 用户没有机会时，需要先完成任务获得机会才能解冻。
- 冰冻和解冻都必须生成事件记录及机会流水。

前端按钮文案：`消耗 1 次机会解冻`。

### 6.2 炸弹

- 用户落在炸弹格后，立即回到当前圈的第 0 格。
- 已经完成的圈数不减少，只清空当前圈的格子进度。
- 本次消耗的跳棋机会不退还。
- 到达当前进度的时间更新为炸弹事件处理完成时间。

例如用户完成 3 圈、当前位于第 20 格，触发炸弹后变为完成 3 圈、当前第 0 格。

### 6.3 奖励格

- 用户落在奖励格后生成中奖记录，并按奖品类型发放奖励。
- 用户以后再次落在同一格，可以再次中奖，但仍受库存和个人上限限制。
- 中奖记录应保存圈数、格子、奖品配置快照、发放状态和关联流水。
- USDT、PHP、VND 等资金类奖励建议采用“先中奖、后发放”的处理流程。

```text
待发放 → 发放中 → 已发放
                  → 发放失败
```

### 6.4 前进和后退格

- 落到“前进 3 格”或“前进 5 格”后立即移动对应格数，不额外消耗机会。
- 前进移动可以越过终点；越过时正常增加完成圈数，并继续计算剩余格数。
- 落到“后退 2 格”后立即后退 2 格，不额外消耗机会。
- 后退最多退到当前圈第 0 格，不扣减已经完成的圈数。
- 为避免连续事件导致流程复杂或无限循环，每次跳棋只触发第一次落脚格事件；事件移动后的目标格不再触发第二次奖励或事件。
- 移动记录应同时保存骰子落脚位置和事件处理后的最终位置，前端先播放骰子移动，再播放事件移动。

## 7. 实时排行榜

### 7.1 排名规则

排行榜按以下条件依次比较：

1. 已完成圈数越多，排名越靠前。
2. 圈数相同时，当前格子数越大，排名越靠前。
3. 圈数和格子数相同时，越早到达当前进度，排名越靠前。
4. 到达时间完全相同时，用户 ID 较小者靠前，以保证排名唯一。

参考查询：

```sql
SELECT user_id, completed_laps, current_position, progress_reached_at
FROM activity_users
WHERE activity_id = :activity_id
  AND ranking_status = 'eligible'
ORDER BY completed_laps DESC,
         current_position DESC,
         progress_reached_at ASC,
         user_id ASC
LIMIT 20;
```

用户有效棋盘位置变化时更新 `progress_reached_at`；签到、充值或领取机会不会修改该时间。

### 7.2 展示内容

- 当前排名。
- 脱敏昵称和用户 ID。
- 已完成圈数。
- 当前格子数。
- 累计进度。
- 到达当前进度的时间。
- 对应排名奖励。
- 当前登录用户自己的排名和进度。

榜单公开展示前 20 名。如果当前用户不在前 20 名，页面底部单独展示其实际排名。

### 7.3 最终排名奖励

| 最终排名 | 奖励 |
| --- | --- |
| 第 1 名 | iPhone 17 Pro |
| 第 2 名 | 500 USDT |
| 第 3 名 | 400 USDT |
| 第 4 名 | 300 USDT |
| 第 5 名 | 200 USDT |
| 第 6～10 名 | 每人 100 USDT |
| 第 11～20 名 | 仅展示排名，无排名奖励 |

活动结束后应立即冻结榜单。后台完成作弊和充值订单审核后，将榜单从“待确认”变为“已确认”，再发放奖励。第二名奖励仅为 500 USDT，不附加其他奖品。

## 8. 用户端页面设计

### 8.1 活动首页

展示内容：

- 活动名称、规则入口和结束倒计时。
- 当前剩余跳棋机会。
- 用户当前圈数、格子数及排名。
- 36 格可视化棋盘和棋子。
- 跳棋或解冻按钮。
- 最近中奖播报。
- 任务列表、机会明细、中奖记录和排行榜入口。

按钮状态：

| 用户状态 | 按钮文案 |
| --- | --- |
| 正常且有机会 | 立即跳棋 |
| 已冰冻且有机会 | 消耗 1 次机会解冻 |
| 没有机会 | 去做任务 |
| 活动未开始 | 活动即将开始 |
| 活动已结束 | 活动已结束 |

### 8.2 任务列表

| 任务 | 状态展示 | 奖励 | 操作 |
| --- | --- | ---: | --- |
| 每日签到 | 未签到、已签到、连续天数 | 5 或 10 次 | 签到 |
| 本人累计充值 | 累计金额、距下一档金额 | 每 10U 获得 10 次 | 去充值 |
| 邀请好友注册 | 已邀请有效人数 | 每人 5 次 | 去邀请 |
| 好友首次充值达标 | 已达标人数、待达标人数 | 每人 10 次 | 查看邀请 |

签到任务应提示未来奖励，例如：`已连续签到 6 天，明天签到可获得 10 次跳棋机会`。

### 8.3 机会明细

每条记录显示发生时间、来源、增加或扣除数量、变化后余额、关联业务及状态。

典型来源：

```text
每日签到                  +5
连续签到第 7 天          +10
本人充值                 +10
邀请好友注册              +5
好友首次充值达标          +10
正常跳棋                  -1
冰冻解冻                  -1
奖励格赠送                +N
管理员补发                +N
退款或风控扣回            -N
```

### 8.4 中奖列表

分为“我的中奖记录”和“全站中奖播报”。记录展示中奖时间、所在圈数和格子、奖品名称、数量及发放状态。全站播报中的昵称、手机号、邮箱等信息必须脱敏。

### 8.5 排行榜页面

展示前 20 名、当前用户实际排名、榜单最后更新时间、排名计算规则和最终奖励说明。

## 9. 运营后台

后台至少包含以下模块：

1. **活动配置**：时间、时区、棋盘格数、充值比例、签到及邀请奖励。
2. **棋盘配置**：每个格子的类型、奖品、库存、个人上限和启停状态。
3. **用户活动状态**：圈数、格子、冰冻状态、剩余机会和排名资格。
4. **机会流水**：查询所有获得、消耗、补发和扣回记录。
5. **中奖管理**：审核和发放资金、VIP、电池等奖励。
6. **邀请关系**：查询邀请绑定、注册奖励和好友首充奖励。
7. **排行榜管理**：实时榜、冻结榜、最终榜及取消资格操作。
8. **风控管理**：根据 IP、设备、账号、邀请关系和充值订单识别异常行为。
9. **人工操作**：补发机会、补发奖品、冻结用户和取消排名资格。
10. **审计日志**：记录管理员、操作时间、原因及操作前后数据。

## 11. PHP 后端设计

### 11.1 推荐技术栈

- PHP 8.2 或以上，生产环境优先使用仍在安全维护周期内的稳定版本。
- Laravel 12，项目锁定主版本，避免部署时自动升级到更高主版本。
- SQLite，开启 WAL 模式，适合本项目低访问量和单服务器部署场景。
- Laravel Session 或 Sanctum 实现登录鉴权。
- Laravel Scheduler 处理活动启停、榜单冻结和库存检查。
- Queue 处理异步派奖、通知和排行榜快照。
- 前端可使用 Vue 3，也可以首版使用 Laravel Blade。
- Node.js 只用于开发或部署阶段构建前端静态资源；完成 `npm run build` 后，PHP 运行环境不需要常驻 Node.js 服务。
- Nginx + PHP-FPM 部署。

版本选择说明：

- PHP 8.5 是开发机器当前安装的 PHP 版本，不代表项目必须使用 PHP 8.5。
- 项目的最低目标版本固定为 PHP 8.2，开发代码及 Composer 依赖必须兼容 PHP 8.2。
- 不建议将 PHP 7.4 作为最低版本。PHP 7.4 以及支持它的 Laravel 8 均已停止安全维护，而本活动涉及登录、充值和奖励发放，不应使用停止维护的运行环境。
- 如果现有服务器只支持 PHP 7.4，应优先升级服务器到 PHP 8.2，再部署本项目，而不是降低框架版本。
- Laravel 11 在当前实施日期已结束安全维护，并且 Composer 会因相关安全公告阻止安装，因此实际实现升级为仍受支持的 Laravel 12。
- `composer.json` 应限制 PHP 为 `^8.2`、Laravel Framework 为 `^12.0`，并提交 `composer.lock`，保证开发、测试和生产环境安装相同依赖版本。

### 11.2 一次跳棋的事务流程

PHP 后端必须在同一个数据库事务内完成：

1. 校验登录状态和活动有效期。
2. 校验用户参与资格与排名资格。
3. 校验客户端生成的唯一 `request_id`，重复请求直接返回原结果。
4. 锁定或原子更新用户活动状态。
5. 检查机会余额。
6. 原子扣除 1 次机会并写入流水。
7. 如果用户已冰冻，执行解冻、保存记录并结束本次操作。
8. 使用服务端安全随机数生成 1～6 点。
9. 计算新的圈数和格子位置。
10. 读取目标格配置并执行奖励、前进、后退、冰冻或炸弹事件。
11. 创建中奖记录或奖励流水。
12. 更新用户进度和 `progress_reached_at`。
13. 保存完整移动记录。
14. 提交事务。
15. 返回旧位置、点数、移动路径、新位置和事件结果，供前端播放动画。

任何一步失败都应回滚，不得出现扣除机会但未移动、发奖但未记录等不一致状态。

### 11.3 SQLite 适用性结论

**本项目用户访问量不大，使用 SQLite 是合适的。** 它部署简单、不需要单独维护数据库服务，事务能力也足以保证机会扣除、棋盘移动和中奖记录的一致性，适合单台服务器上的首版活动。

建议在满足以下条件时采用 SQLite：

- PHP 应用主要运行在一台服务器或共享同一块本地磁盘上。
- 日常同时进行签到、充值回调和跳棋的写请求不多。
- 数据库文件放在可靠的本地磁盘，不放在 NFS 等网络文件系统上。
- 定时备份数据库文件，并对重要派奖流水保留审计记录。

需要注意：访问量低不等于永远没有瞬时并发。用户可能在活动开始、整点或截止前集中签到和跳棋，因此仍应正确处理写锁、事务、幂等和重试。

出现以下情况时，应迁移到 MySQL 或 PostgreSQL：

- 应用需要多台服务器同时读写同一个数据库。
- 经常出现 `database is locked`，优化短事务后仍无法解决。
- 活动产生明显的集中写入高峰，写请求需要排队较长时间。
- 需要复杂的数据分析、读写分离、在线扩容或高可用数据库集群。

### 11.4 SQLite 配置与并发注意事项

虽然本项目可以使用 SQLite，开发时仍应做到：

- 启用 `PRAGMA journal_mode=WAL`。
- 配置合理的 `busy_timeout`。
- 写事务尽可能短，事务内不调用外部支付、消息或钱包接口。
- 对榜单和业务查询建立完整索引。
- 排行榜可从用户活动状态表查询，并定时生成快照，避免扫描全部流水。
- 资金奖励先落库，事务提交后再通过队列调用外部钱包。
- 数据层保持标准 SQL 和 Repository/Service 隔离，为以后迁移 MySQL 或 PostgreSQL 留出空间。
- 上线前通过压测模拟活动高峰；如果锁等待和接口延迟在可接受范围内，即可继续使用 SQLite。

## 13. 风控与安全

- 登录、签到、邀请、跳棋和充值回调接口分别设置限流策略。
- 服务端生成随机点数，不接受前端传入结果。
- 邀请奖励使用唯一业务键，确保只发放一次。
- 对相同设备、IP、手机号、邮箱、钱包地址和支付账户进行关联分析。
- 排名前 20 名和大额中奖用户进入人工复核名单。
- 充值退款、拒付和异常订单需要自动关联并处理已发机会。
- 后台人工调整必须填写原因并保存操作前后值。
- 用户隐私数据在页面、日志及中奖播报中必须脱敏。
- 定期备份 SQLite 数据库，并演练恢复流程。

## 14. 测试验收重点

- 第 7、14 天签到正确发放 10 次；断签后正确从第 1 天开始。
- 充值 25U 后再充值 5U，分别正确发放 20 次和 10 次。
- 同一充值回调重复到达时不会重复发放。
- 好友分多次累计充值到 10U 时，只给邀请人发放一次 10 次机会。
- 好友后续充值不会再次给邀请人发放，但好友本人正常获得机会。
- 第 34 格抽到 5 点时正确进入下一圈第 3 格。
- 前进 3 格和前进 5 格可正确跨圈，并且不额外扣除机会。
- 后退 2 格最多退到当前圈起点，不减少已完成圈数。
- 事件移动后的目标格不会再次触发奖励或事件。
- 冰冻后消耗一次机会只解冻、不移动。
- 炸弹只清空当前圈进度，不减少已完成圈数。
- 并发点击跳棋按钮时只成功执行一次。
- 排名按圈数、格子数、到达时间和用户 ID 正确排序。
- 活动结束后不能继续签到、领取活动机会或跳棋。
- 排行榜冻结后，最终排名不会因后续数据变动而改变。

## 15. 已确定的业务口径

- “抽奖机会”在活动页面统一称为“跳棋机会”。
- 每次正常跳棋随机前进 1～6 格。
- 普通签到每天 5 次；连续签到第 7、14 天当天共 10 次。
- 连续签到中断后从第 1 天重新计算。
- 邀请有效好友注册，邀请人获得 5 次。
- 好友累计首次达到 10U，邀请人获得 10 次且只发一次。
- 本人每累计充值 10U 获得 10 次，可以重复获得。
- 炸弹只清除当前圈的格子进度，不清除已经完成的圈数。
- 棋盘采用大富翁式环形布局，普通格和事件格为主，奖品格只占少部分。
- 前进、后退事件不额外消耗机会，且事件移动后的目标格不触发连锁事件。
- 页面同时适配 PC 和手机端，手机端无需横向滚动即可看到完整棋盘。
- 排行榜展示前 20 名，最终第 1～10 名获得排名奖励。
- 第 1 名为 iPhone 17 Pro，第 2 名仅为 500 USDT。
