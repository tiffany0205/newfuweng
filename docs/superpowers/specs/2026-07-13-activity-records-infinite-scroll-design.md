# 活动记录默认展开与滑动分页设计规格

## 目标

活动页底部的“机会明细”和“中奖列表”默认同时展开。每个列表首次只展示最新 10 条，用户滑动到对应列表底部时，独立加载下一批 10 条，直到全部记录加载完成。

## 已确认交互

- 两个 `<details>` 都带 `open`，进入页面即可查看内容。
- 两个列表各自首批展示 10 条，不共享页码或加载状态。
- 页面滚动使某个列表的加载哨兵进入视口时，自动加载该列表下一批 10 条。
- 每批数据追加到现有表格底部，最新记录始终在最上方。
- 加载期间显示“加载中…”，加载结束显示“已加载全部”。
- 请求失败显示“加载失败，点击重试”；重试只影响失败的列表。
- 按钮可以被键盘聚焦和点击，作为自动加载之外的无障碍与弱网络回退入口。

## 分页接口

新增两个仅限登录用户访问的 GET 接口：

- `/activity/records/chances`
- `/activity/records/winnings`

请求参数：

- `cursor`：上一批最后一条记录的整数 ID；首批记录由活动页服务端渲染，后续请求必须携带该游标。
- `limit`：不对客户端开放，服务端固定为 10，防止一次请求拉取过多数据。

响应结构：

```json
{
  "data": [],
  "next_cursor": 123,
  "has_more": true
}
```

机会明细每条数据包含 `id`、`created_at`、`remark`、`amount`和`balance_after`。中奖记录每条数据包含 `id`、`created_at`、`prize_name`和本地化后的`status_label`。

## 查询规则与安全

- 查询必须同时限定当前活动 ID 和当前登录用户 ID，禁止读取其他用户记录。
- 按 `id DESC` 排序，后续查询使用 `id < cursor`，避免加载期间新增记录导致 offset 分页出现重复或跳项。
- 每次查询取 11 条：返回前 10 条，第 11 条仅用于判断`has_more`。
- 游标必须是大于 0 的整数；无效游标返回 422。
- 没有更多数据时返回空数组、`next_cursor: null`和`has_more: false`。

## 首屏渲染

- `GameController@index`分别查询两个列表的前 11 条。
- Blade 只渲染前 10 条，并把第 10 条 ID 作为各自的初始游标。
- 如果总数不超过 10 条，直接显示“已加载全部”，不发起后续请求。
- 空列表保持现有“暂无记录”文案，不显示可继续加载状态。

## 前端加载器

- 每个记录区使用统一的`data-infinite-records`结构，声明接口 URL、当前游标和是否还有数据。
- 一个通用初始化函数为两个区域分别维护`loading`、`cursor`和`hasMore`。
- `IntersectionObserver`观察各自哨兵；不支持该 API 时保留“加载更多”按钮供用户点击。
- 请求进行中禁止重复触发。
- 追加前按记录 ID 去重，防止网络重试或重复回调插入相同行。
- 所有文本通过 DOM API 写入，不拼接来自接口的未转义 HTML。

## 展示与响应式

- 继续使用现有双列表桌面两列、手机单列布局。
- 列表本身不设置内部滚动条，使用页面自然滚动，避免手机端嵌套滚动卡顿。
- 加载状态位于各自表格下方，保持足够点击面积和清晰对比度。
- 动态状态区域使用`aria-live="polite"`，不会打断用户当前操作。

## 测试与验收

- 功能测试断言两个 `<details>` 默认均带`open`且首屏最多各 10 条。
- 接口测试验证固定每批 10 条、倒序、正确游标和`has_more`。
- 接口测试验证用户隔离，其他用户的机会与中奖记录绝不返回。
- 接口测试验证无效游标返回 422，末页返回完成状态。
- 浏览器测试验证两个列表独立加载、无重复行、失败可重试和手机端无横向溢出。
- 运行完整 PHPUnit、Pint、Vite 构建、Composer audit 与`git diff --check`。

## 不在本次范围

- 不修改机会流水或中奖记录的生成逻辑。
- 不为管理员页面或幸运中心列表增加无限滚动。
- 不记忆用户手动折叠后的状态；刷新页面后两个列表仍默认展开。
- 不增加 WebSocket 或实时推送，新产生的记录在刷新页面后出现在顶部。
