# 地标数据补偿与实际位置展示设计

## 目标

修复已有 SQLite 数据库升级后，星光广场等格子没有被识别为正式地标的问题；补回受影响用户已经发生但未计入的地标收集和确定性幸运值；同时让每次掷骰的最终位置在主要界面和机会流水中使用一致的第 1～36 格表述。

## 已确认根因

- `2026_07_13_000300_create_landmark_help_tables.php` 只增加了地标相关字段和表，已有 `board_cells` 全部获得默认 `safe` 分类。
- 12 个正式地标的 `category`、`landmark_code`、`effect_code` 只存在于 `DatabaseSeeder`。全新测试数据库正确，但只执行 `migrate --force` 的已有数据库不会得到这些配置。
- 因此已有环境中的星光广场可能仍按普通格结算，不会创建 `user_landmarks`，也不会执行 `lucky_2`。
- 机会扣减流水在掷骰前写入，备注固定为“跳棋消耗”；走棋结束后没有使用点数、最终位置和格子名称更新该流水。
- `current_position` 在数据库中为 0～35，部分页面直接显示原始值，另一些页面显示 `+1`，造成用户看到的格数不一致。

## 方案比较

### 方案 A：只修复未来结算

新增迁移同步 12 个地标，但不处理历史记录。改动最小，但用户已经到达星光广场却丢失的地标和幸运值无法恢复，不符合本次问题的补偿诉求。

### 方案 B：配置同步、历史补偿、位置统一（采用）

新增幂等数据迁移同步正式地标，并从 `board_moves` 的最终位置回放受影响的历史地标访问；补回确定性幸运值，更新历史机会备注。运行时继续正常结算，并统一所有用户可见位置。该方案能同时修复根因、存量数据和后续体验。

### 方案 C：每次请求动态检查并修正棋盘

在控制器中检测错误数据并现场修复。可以绕过部署迁移遗漏，但会把数据治理混入高频业务请求，增加并发和维护风险，因此不采用。

## 数据迁移

新增一条迁移，不修改已经执行过的旧迁移。

### 同步正式地标

按活动与固定 `position` 更新 12 个正式地标。使用位置作为主键语义，不依赖“星光广场/星光大道”等显示名称，兼容已有环境中的文案差异。

每个地标同步：

- `category = landmark`
- `landmark_code`
- `effect_code`
- `description`

同时将所有非地标格按现有 `type` 恢复为 `safe`、`boost`、`reward` 或 `risk`，避免旧数据库继续保留错误的默认分类。地标阶段奖励和 FAQ 使用 `updateOrInsert` 补齐，迁移可安全执行一次且结果确定。

### 补回历史地标收集

以 `board_moves.activity_id + user_id + to_position` 统计历史有效 `move`：

- 只处理迁移前尚无对应 `user_landmarks` 记录的用户与地标，避免重复补偿已正常结算的数据。
- 创建地标记录，`visit_count` 使用历史到达次数，首次和最近到达时间取历史最早、最晚时间。
- 补回每个地标从第二次访问开始应得的基础幸运值 `visit_count - 1`。
- 对 `lucky_N` 地标，按每次历史访问补回 `N` 点；星光广场首次历史到达因此补回 2 点。
- 不追发彩虹随机结果、机会、道具或防护效果，因为这些结果无法从历史数据可靠重建；避免凭空生成随机奖励。

### 修复历史机会流水

通过 `chance_transactions.business_key = move-{request_id}` 关联 `board_moves`，将旧的“跳棋消耗”更新为：

- 正常掷骰：`掷出 X 点 · 到达第 N 格 格子名称`
- 解冻操作：`解冻成功 · 停留第 N 格 格子名称`

其中 `N = to_position + 1`。

## 后续走棋数据流

1. 扣除一次机会并创建流水，保留业务键保证幂等。
2. 完成点数、道具、移动格和最终格结算。
3. 保存 `board_moves`。
4. 使用同一 `request_id` 更新本次机会流水备注为实际点数与最终位置。
5. API 返回 `display_position = to_position + 1`、最终格名称、地标数量、幸运值和本次机会流水。
6. 重复请求命中已有 `board_moves` 时，重新构造相同响应，不再次增加地标或幸运值。

## 用户界面

- 顶部“当前格子”和中央状态统一显示第 1～36 格，不再直接暴露数据库的 0～35。
- 每次走棋完成后，中央事件提示显示 `第 N 格 · 结果说明`。
- 结果弹框的详情显示 `抵达第 N 格 · 格子名称`；解冻显示停留位置。
- 底部机会明细立即插入本次真实流水，无需刷新；按记录 ID 去重，避免与滚动加载重复。
- 游戏页实时排行榜和幸运中心总进度榜显示 1～36 格；排序仍使用数据库 0～35，不改变排名规则。
- 地标结果返回并显示最新地标进度和幸运值；星光广场首次到达显示地标解锁与幸运值 +2。

## 错误处理与兼容性

- 数据迁移在事务支持范围内执行；没有历史走棋数据时只同步配置。
- 找不到对应格子名称时使用“当前位置”，但仍显示实际格数。
- 机会流水更新失败会使整个走棋事务回滚，避免扣机会成功但结果记录不一致。
- 不改变随机点数、奖品概率、圈数、排名排序、音效和弹框触发规则。

## 测试与验收

- 使用不运行 Seeder 的旧库升级场景，证明星光位置会从 `safe` 变为 `landmark`。
- 旧历史首次到达星光位置会补建地标，并增加 2 点幸运值；重复运行/已有记录不会重复补偿。
- 新用户首次和重复到达星光地标分别获得正确地标进度与幸运值。
- 正常掷骰和解冻的机会流水都包含第 1～36 格实际位置。
- API 重复请求仍幂等，并返回相同 `display_position`。
- Blade 与 JavaScript 统一使用用户位置；排行榜只改显示、不改排序。
- PHPUnit、Node 测试、Vite 构建、Pint、界面规范扫描和数据库迁移测试全部通过。
